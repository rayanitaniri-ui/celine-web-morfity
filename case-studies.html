Case Study 01 · Travel & SaaS

Travel Is Exciting — Full-Stack Booking Platform

IndustryTravel & Hospitality
Timeline14 weeks
Team4 engineers + PM

The Challenge

Travel Is Exciting was operating with a patchwork of spreadsheets, a legacy PHP site, and manual booking processes. The team was losing 30% of potential bookings to friction at checkout, and their inventory was managed by phone calls to hotel partners.

They needed a modern platform that could handle real-time inventory from multiple suppliers, process payments in three currencies, and give the operations team live visibility into bookings — all without rebuilding from scratch every 18 months.

Our Solution

We designed a multi-tenant booking platform with a public-facing Next.js frontend, a Node.js API layer, and a PostgreSQL database with row-level security for tenant isolation. Real-time inventory is handled via a queue-based architecture using Redis — reservation locks expire after 12 minutes, releasing inventory back to the pool automatically.

The admin CMS gives the operations team full control over packages, pricing, and availability without engineering involvement. Stripe handles multi-currency checkout with automatic tax calculation for GCC markets.

Development Process

Weeks 1–2: Discovery sessions with the operations team. Mapped every booking workflow. Identified integration points with existing supplier APIs and built a data model that could handle 15+ room types per property.

Weeks 3–10: Four two-week sprints. Sprint 1: core booking flow and payment integration. Sprint 2: admin CMS and inventory management. Sprint 3: supplier API integrations and real-time availability. Sprint 4: reporting dashboard and email automation.

Weeks 11–14: User acceptance testing with the ops team, performance testing under simulated peak load (2,000 concurrent users), SEO audit, and staged rollout alongside the legacy system.

Architecture Overview

Next.js frontend deployed on Vercel with ISR for destination pages. Node.js API on AWS ECS with auto-scaling. PostgreSQL on AWS RDS with read replicas for reporting queries. Redis for session storage and inventory locks. S3 and CloudFront for media delivery. GitHub Actions for CI/CD with blue-green deployments.

Case Study 02 · E-commerce & Analytics SaaS

Merchant Analytics Suite — Real-Time E-commerce Intelligence

IndustryE-commerce
Timeline18 weeks
Team5 engineers + PM

The Challenge

A group of multi-store Shopify merchants were flying blind. Shopify's built-in analytics showed revenue but couldn't answer the question that actually matters for growth: which customer segments are worth acquiring, and which promotions are generating repeat purchasers vs. one-time buyers?

The client needed a SaaS platform that could pull data from multiple Shopify stores, normalise it, and surface cohort-level insights that weren't available anywhere else. And it needed to be fast — founders don't want to wait 20 seconds for a dashboard to load.

Our Solution

We built a multi-tenant SaaS with a React frontend and a Node.js API backed by PostgreSQL for normalised merchant data and a TimescaleDB extension for time-series analytics queries. Shopify store connections are OAuth-based — merchants connect a store in under 60 seconds, and historical data syncs in the background.

The analytics engine runs cohort definitions on pre-aggregated materialized views, keeping query times under 300ms for 90% of dashboard loads. Stripe billing handles subscription management with usage-based seat expansion.

Development Process

Weeks 1–3: Analytics requirements deep-dive with the merchant team. Built a data model that could represent 10+ years of Shopify order history per store. Designed the cohort calculation engine before writing a line of UI code.

Weeks 4–14: Four 2-week sprints. Sprint 1: Shopify OAuth integration and data sync pipeline. Sprint 2: Core analytics queries and materialized views. Sprint 3: Dashboard frontend with chart library. Sprint 4: Billing integration and multi-store management.

Weeks 15–18: Beta with 5 pilot merchants. Identified three missing cohort definitions based on real usage. Performance tuning under real data volumes (2M+ orders). Public launch with onboarding flow.

Architecture Overview

React SPA hosted on CloudFront. Node.js API on AWS ECS. PostgreSQL + TimescaleDB on RDS for analytics. Redis for query result caching. BullMQ for async Shopify data sync jobs. Stripe for subscription billing with webhook processing.

Case Study 03 · AI & Automation

AI Studio — No-Code AI Workflow Automation Platform

IndustryAI / SaaS
Timeline20 weeks
Team6 engineers + PM

The Challenge

The client wanted to compete in the AI automation space — but their target customers were non-technical teams inside regional enterprises who couldn't write prompts or configure APIs. Every existing tool required too much setup to be adopted by an operations manager or marketing coordinator.

The challenge: build an AI workflow builder with a drag-and-drop interface simple enough for non-engineers, but powerful enough to handle complex multi-step automations with real business logic, conditional branching, and integration with existing enterprise tools.

Our Solution

We built a node-based workflow editor on top of React Flow, backed by a Python execution engine that processes workflows asynchronously. Each workflow is serialised as a DAG (Directed Acyclic Graph) and stored in PostgreSQL. Execution runs on isolated Docker containers per workflow run to prevent cross-tenant contamination.

LLM calls route through our abstraction layer supporting OpenAI, Anthropic, and Gemini — so customers aren't locked to one provider. Enterprise integrations (Salesforce, HubSpot, Slack, WhatsApp Business) are handled via a connector library using OAuth2.

Development Process

Weeks 1–3: Competitor analysis and user research with 8 target customers in the UAE and Lebanon. Defined the minimum node library: 12 node types covering AI generation, data transformation, API calls, conditionals, and loop handling.

Weeks 4–16: Five 2-week sprints. Sprint 1: Workflow editor canvas and node system. Sprint 2: Python execution engine and async job processing. Sprint 3: LLM abstraction layer and first integrations. Sprint 4: Execution logs, debugging UI, and error handling. Sprint 5: Enterprise SSO, team management, and usage limits.

Weeks 17–20: Security audit (execution isolation, prompt injection testing). Load testing execution engine at 500 concurrent workflow runs. Soft launch with 12 enterprise pilot customers.

Architecture Overview

Next.js frontend with React Flow canvas. Python FastAPI execution engine on AWS ECS with Docker isolation per workflow run. PostgreSQL for workflow storage. Redis for execution state and job queues. Celery for async task processing. CloudFront for CDN. Terraform for infrastructure as code.
